# AI Agent 身份层调研

> 调研日期：2026-03-11
> 来源：多轮 WebSearch + 深度分析

## 一、历史方案及其失败原因

### PGP / Web of Trust（1991）
- 核心：去中心化信任网络，互相签名建立信任链
- 失败原因：可用性灾难（密钥管理极其复杂）、冷启动问题、信任图谱最多2跳、密钥服务器被污染

### Keybase（2014-2020）
- 核心：加密身份 + Social Proof（在 Twitter/GitHub 发布签名声明证明身份关联）
- 做对了的：Social Proof 模型本身是正确的——把公钥跟已有社交身份绑定
- 失败原因：服务器端闭源形成中心化依赖、VC 融资要求快速增长与基础设施长期稳定性矛盾、2020年被 Zoom acqui-hire 后产品废弃
- 核心教训：**身份基础设施作为 VC 驱动的创业公司天然矛盾**

### W3C DID（2022）
- 核心：自发行标识符格式（did:method:identifier），多种 DID 方法
- 困境：120+ 种方法碎片化、Apple/Google/Mozilla 集体反对、did:web 本质不比传统 PKI 更强、缺乏激励机制
- 教训：**"standardize first, adopt later" 路线失败**

### SSI / Verifiable Credentials
- 核心：用户持有数字凭证，选择性披露，发行者/持有者/验证者三方分离
- 困境：认知门槛高、行为迁移成本大、鸡生蛋问题

### 五大根本失败原因（跨方案共性）
1. **冷启动 + 鸡蛋问题**：没人用就没有价值
2. **用户无痛点感知**：Google/Apple 登录"够用了"
3. **可用性输给中心化**：PGP 密钥管理远不如"微信扫一扫"
4. **商业模型矛盾**：基础设施需要长期稳定，VC 需要快速退出
5. **标准碎片化**：DID 120+ 方法，SSI 多种凭证格式

## 二、AI Agent 改变的条件

### 改变了的
1. **用户变了**：身份用户从人类（讨厌管密钥）变为 Agent（天然擅长密码学）→ **可用性问题消失**
2. **痛点变了**：Agent-to-Agent 交互没有人在循环中判断 → **从"可选"变"必须"**
3. **规模频率变了**：Gartner 预测 2026年 40% 企业应用嵌入 Agent → **高频自动**
4. **委托链出现**：人→Agent→子Agent 需要可验证证据链
5. **监管需求**：Agent 自主操作需要审计链

### 没改变的
1. 鸡蛋问题依然存在
2. 碎片化可能更严重（MCP vs A2A vs 各家自建）
3. 信任锚点运营的中心化需求

## 四、关于信任锚点的分析

### "不依赖平台"的真正含义
不是"不依赖任何东西"，而是**信任锚应该是可替换的、多元的、自主可控的**。

### 信任锚选项对比
| 锚点 | 自主性 | 持久性 | 易用性 |
|------|--------|--------|--------|
| GitHub Profile | 低（依赖微软） | 中 | 高 |
| 个人域名 DNS TXT | 高（你拥有域名） | 高 | 中 |
| 个人网站 .well-known | 高 | 高 | 中 |
| 区块链 | 最高 | 最高 | 低 |
| IPFS | 高 | 依赖 pin | 低 |

### 核心原则
1. 多锚点：不绑死一个平台
2. 自有域名优先：DNS 是最接近"自主"的信任基础
3. 锚点可迁移：平台倒了可以重建
4. **密钥是根，锚点是叶**：私钥才是身份根，外部锚点是发现和验证手段

## 五、信任 vs 隐私的核心张力

- 信任要求身份可验证（公开性）
- 隐私要求身份不可关联（隐蔽性）
- 两者天然矛盾
- MVP 策略：**先做信任/公开，后扩展隐私/不可关联**
- 技术路径：从"公开身份"派生"匿名子身份"是成熟技术（HD 钱包、BIP-32）
